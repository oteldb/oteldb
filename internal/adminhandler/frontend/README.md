# @oteldb/admin-frontend

Full-featured web UI for the oteldb admin panel — TypeScript + React + Vite,
with a typed API client generated by [Orval](https://orval.dev) from the admin
OpenAPI spec. Uses [Bun](https://bun.sh) as the package manager/runtime.

The built SPA (`dist/`) is embedded into the Go binary by
`internal/adminhandler/ui.go` and served at the admin bind address (`:8090` by
default), same-origin with the `/api/v1/*` endpoints.

## Stack

- **Bun** — install + run scripts.
- **Vite** — dev server and production bundler (relative `base` for embedding).
- **Orval** — generates TanStack Query hooks + models from `_oas/admin.yml`
  into `src/api/` (regenerated on every build; do not edit by hand).
- **TanStack Query** — data fetching, caching, live polling.
- **React Router** (hash mode) — client-side routing without server rewrites.
- **Recharts** — the live heap chart on the Runtime page.

The theme is a single dark palette (`src/theme.css`); there is intentionally no
light variant.

## Develop

```bash
bun install
# Run oteldb locally so /api is reachable (vite proxies /api → :8090):
bun run dev            # http://localhost:5273
```

## Regenerate the API client

```bash
bun run gen            # orval → src/api/{admin.ts,model/*}
```

The spec lives at `../../../_oas/admin.yml` (repo `_oas/admin.yml`) and is the
same source the ogen server is generated from, so client and server stay in
lock-step.

## Build (required before `go build` picks up UI changes)

```bash
bun run build          # orval + tsc + vite build → dist/
```

`dist/` is committed so the Go build embeds the UI without needing a Bun
toolchain. After a UI change: `bun run build`, then rebuild the binary
(`go build -o ./oteldb ./cmd/oteldb`).

## Requests

All hooks route through the Orval mutator in `src/lib/fetcher.ts`, which adapts
requests onto native `fetch`, keeps them same-origin, and surfaces the admin
API's structured `{ error_message }` body as a thrown `ApiError`.
